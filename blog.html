Strict Equality vs Loose Equality- 01/10/2017

Today lets talk about two types of JavaScript equalities. The first of the two equalities is known as strict equality aka the triple equal operator which shows up in our code as ===. The second of the two equalities know as loose equality aka the double equal operator shows up in our code as ==.


When comparing the equality of primitive JavaScript data types such as Booleans, Numbers, Strings, Undefined and Null both == and === will return true in the following examples, (true == true), (true === true), ('yes' == 'yes'), ('yes' === 'yes'), (1 == 1), (1 === 1), (undefined == undefined) and (undefined === undefined).


As demonstrated above with primitive data types the === and == are comparing value. When the comparisons see the same exact value they return a Boolean value of true.


Things change a bit when we start comparing different primitive values with ==, something called type coercion takes place. All of the following examples will return a Boolean value of true during comparison, ('1' == 1), (true == 1), ([] == 0) and ('' == 0). This type coercion happens because the interpreter evaluates a string '1' and the number 1 as a truthy value and returns true.


It is important to note that all of these examples shown in the previous paragraph would return false when using ===. It returns false because the interpreter is checking to see if the two comparators are the same exact value, no type coercion takes place. This is why it is a best practice to pretty much always use strict equality.


Things change quite a bit when we move away from primitive data types and focus on object data types. Object data types include of course normal objects as well as arrays and functions. Keep in mind that arrays are array like objects and functions are really just objects that have the special ability to be called/invoked.


Now when we compare object data types with === and == the interpreter is looking to see if those two values are stored in the exact same place in memory. All of the following examples with return a Boolean value of false ([] == []), ([] === []), ({} == {}), ({} === {}), ( (function(){}) == (function(){}) ) and ( (function(){}) === (function(){}) ).


Even though the values shown immediately above are the same, they are object data types and therefore are stored at different places in memory. Even if an object, array or function has they same exact value as another object data type, it will always be stored at different places in memory, thus always returning false when using both == and ===. Thank you for taking the time to read this article.


The JavaScript Keyword 'This' - 01/03/2017

In my early days of studying JavaScript fundamentals, I remember seeing the keyword ‘this’ for the first time while building constructor functions in the pseudo-classical instantiation pattern. Even after reading up on the subject I was still a bit confusing so I reached out to my coding buddy enquiring about its mystical powers. He mentioned that the keyword ‘this’ contains a lot of ‘gotchas’ and made the point that it would only become clearer with further study and practice.


Now after said study and practice I know that the keyword ‘this’ is an auto-bound identifier intended to refer to an object predominantly used during method or constructor function invocation. I also learned that there are five ways that the keyword ‘this’ can appear in your running code.


The first and most common way to use the keyword this is during method invocation where ‘this’ is bound to the object that was left of the “call time dot”.


The second and third ways to use the keyword ‘this’ are inside the call and apply methods. The object passed into the first argument of their parameter list is what the keyword ‘this’ is bound to.


The fourth and fifth binding patterns for the keyword ‘this’ are not useful as they bind themselves to the global object in the browser window. Examples would be when the keyword ‘this’ is used in the global scope or when it is used in a free function invocation. Thank you for reading my post.
